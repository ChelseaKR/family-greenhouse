# Data Protection Impact Assessment (DPIA) — family-greenhouse

Instantiates `STANDARDS/RESPONSIBLE-TECH-FRAMEWORK.md` §C (privacy & data-protection audit).
Closes the tracked gap in `docs/RESPONSIBLE-TECH-AUDITS.md` §C (remediation P2-1).

**System:** Family Greenhouse — a shared household plant-care app. Households of
2–6 people (plus optional no-account "sitters") track plants, care tasks, and
completions. Backend: AWS Lambda + a single DynamoDB table + S3 for photos;
auth via Cognito. Every claim below cites the code or Terraform that implements it.

**Assessor:** repository owner (solo maintainer). **Date:** 2026-07-13.

### What is NOT collected

- No payment card numbers (Stripe-hosted checkout only).
- No precise/device geolocation — household location is a user-typed city, opt-in, off by default.
- No contacts, no address book, no advertising identifiers, no third-party ad trackers.
- No session replay (deliberately not using posthog-js; see `frontend/src/services/analytics.ts` header).
- No plaintext API keys at rest (scrypt hash only).
- No sitter identity — sitters never create an account; the link label is enforced non-PII.
- No HTML email (plain text only — `backend/src/services/emailNotifier.ts`), so no tracking pixels.

## 2. Lawful basis and consent reality

- **Contract** — core service data (account, household, plants, tasks): processing is what the user signed up for. Documented in `docs/compliance.md` §3.
- **Consent (explicit, opt-in)** — SMS phone number (verification code required before any send, `notificationPrefs.ts`; production SMS additionally gated behind the TCPA prerequisites in `docs/compliance.md` §4 and `SMS_NOTIFICATIONS_ENABLED`), household location, pest alerts (off by default "because we want explicit consent before adding to the notification volume"), browser push (browser permission prompt + stored intent).
- **Legitimate interest** — 30-day security audit log (login failures, rate-limit trips) and rate-limiting IP handling.
- **Household consent reality (honest):** this is a _shared_ account model. A member's plant activity, display name, vacation window, and chat messages in shared threads are visible to every other household member by design. Joining via invite is the consent act for that visibility; the roster deliberately hides member emails from other members (`PublicHouseholdMember`, `backend/src/models/types.ts` lines 38–45). Default-on email reminders rest on the service-contract basis; every other channel is opt-in.
- **Subject rights:** self-service export (`GET /me/export`) and erasure (`DELETE /me`) — `backend/src/handlers/me/handler.ts`. Deletion refuses (rather than orphaning a household) when the caller is the lone admin of a multi-member household.

## 3. Threat surfaces (specific people, not abstract attackers)

1. **Shared-household access / abusive-member model.** Every member sees all plants, activity, and — critically — **vacation windows** (who is away from home, and when). A member who leaves or is removed loses access, but data they already saw is unrecoverable. Mitigations: role model (admin vs member), member removal, email hidden from roster. Residual: an abusive co-member is inside the trust boundary by design; the app should never be the only place a vulnerable user records absence dates.
2. **Notification leakage.** Reminder pushes/SMS render on lock screens; content is plant/task-scoped (no member emails), and per-user DND windows exist (`notificationPrefs.ts`). SMS is capped at 140 chars and disabled in production until TCPA controls exist.
3. **Sitter-link forwarding.** A sitter URL is a bearer credential. Bounded: 256-bit CSPRNG token, ≤60-day creator-set window enforced on _every_ public call, revocable, generic 404/410 so the endpoint is not a token oracle, non-PII label (`sitterService.ts`).
4. **Photo URL exposure.** Image URLs are unguessable (UUID key segment) but **bearer-access**: anyone holding the CloudFront URL can fetch the photo — there are no signed URLs or auth on image GET (`publicImageUrl`, `handlers/plants/handler.ts`). Plant photos can incidentally capture home interiors. Open item below.
5. **Invite-code leakage.** 128-bit code, 7-day TTL, single household scope.
6. **Model/API egress.** Chat text and leaf/identification photos leave for Bedrock/Plant.id only on explicit user action; every live and replayed chat-tool result passes through the recursive denylist sanitizer in `backend/src/services/chat/tools.ts` before it reaches the model. No training use is granted by these APIs' standard terms — verify on renewal (open item).
7. **Logs.** Audit lines carry actorEmail + IP by design (bounded set, 30-day retention); application logs are structured and the no-PII discipline is unit-tested for the chat redactor.

## 4. Minimization & purpose-limitation check

- Email stored once (Cognito), mirrored not duplicated in prefs (`notificationPrefs.ts` header).
- Phone: empty string unless SMS opted in; changing it clears `phoneVerified`.
- Analytics carries no PII on either rail; DNT honored in the browser.
- Chat history auto-expires at 30 days rather than accreting.
- Sitter view exposes due tasks only — "the token grants exactly one household's due-task view + completion, nothing else."
- Location is a feature-gated opt-in with a stated reason.
