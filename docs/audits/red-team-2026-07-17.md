# Red-team exercise — chat tool layer (2026-07-17)

**Scope:** the plant-care Bedrock chat feature (`backend/src/services/chat/`),
specifically the prompt-injection surface where **user-controlled fields**
(plant names, notes, locations, custom task labels) and **tool arguments** flow
into tool results, the next model prompt, and the RAG grounding context.

**Standard:** `docs/standards/AI-EVALUATION-STANDARD.md` §2 (red-team). This
document is the first structured red-team artifact for this repo; it closes the
`docs/audits/red-team-<date>.md` gap noted in `docs/RESPONSIBLE-TECH-AUDITS.md`
**at the offline/mechanical layer only** — the live-model layers remain waived
(see "Not run" below).

**Owner:** CKR · **Cadence:** re-run on any `BEDROCK_CHAT_MODEL_ID` swap,
system-prompt rewrite, or new tool added to `TOOL_REGISTRY`.

---

## What was run (deterministic, offline, CI-gated)

An injection corpus — `evals/redteam/injection-corpus.json`, 9 payloads mapped
to OWASP LLM Top-10 categories — is exercised by
`backend/tests/unit/services/chatRedteamInjection.test.ts` (11 test cases,
green in the normal backend suite). Each payload is stored in a user-controlled
field or passed as a tool argument, then run through the **real** tool executors
and the **real** model-boundary sanitizer (`sanitizeToolResultForModel`). No
model is called; the assertions are on the tool layer's mechanical behavior.

### Invariants asserted

| Invariant           | OWASP                                                        | What the test proves                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **household-scope** | LLM06 (excessive agency)                                     | Read tools scope DynamoDB access by `ctx.householdId` (the authenticated caller), never by a value in the tool input. A forged `householdId` argument (`hh-ATTACKER`) and handlebars-style payloads in a `location` field do not change which household is queried — `getPlants` is called with the victim household only.                                                                                                                                          |
| **pii-redaction**   | LLM01 (prompt injection) / LLM02 (sensitive-info disclosure) | `sanitizeToolResultForModel` strips every forbidden key (`email`, `sub`, `userId`, `householdId`, `createdBy`, `assigneeName`, …) before a tool result reaches the model, even when sibling **values** carry injection text. An attacker cannot smuggle a member email/sub to the model by wrapping it in instruction text; and `list_household_plants` output for an injection-named plant contains neither the owner's `createdBy` id nor the household id.       |
| **write-gate**      | LLM06                                                        | `propose_reminder_task` never writes; it validates and returns a proposal card. A cross-household/hallucinated `plantId` is rejected (`getPlant` is scoped to `ctx.householdId` → null), a non-member `assignedTo` is rejected, and a schema-violating `customType` (>50 chars, injection payload) is rejected — coercion text in adjacent fields does not bypass the checks. A legitimate proposal for an injection-named plant still emits zero PII to the model. |

### Result

**All 11 cases pass.** No injection payload stored in a user-controlled field
was able to (a) widen a tool's household scope, (b) exfiltrate a household
member's PII to the model prompt, or (c) coerce a write past the confirm-card
validation. These are the mechanical guardrails the architecture already
committed to (`tools.ts` PII denylist + `ctx.householdId` scoping +
propose-not-write); this exercise turns that commitment into a regression gate.

## Not run (honesty boundary — still waived)

This offline exercise **cannot** assert anything about the model's own behavior,
because it does not call Bedrock. The following remain **not built** and are
covered by the dated AI-eval waiver (`docs/RESPONSIBLE-TECH-AUDITS.md`, expires
2026-10-05):

- **Live-model refusal / no-fabrication under injection.** Whether the model,
  when it actually reads an injected instruction inside a tool result, declines
  to follow it (rather than, say, echoing a fabricated email it invents) is
  untested here. That needs the live generation-layer eval — a ⛔USER-triggered,
  budget-capped dispatch job against real Bedrock — which does not exist.
- **Promptfoo OWASP LLM01–10 config against the mock chat path.** Not built.
  The mechanical invariants above cover the _data-layer_ consequences of LLM01
  and LLM06; a full Promptfoo run would additionally probe the _model's_
  response text across all ten categories.
- **Garak baseline** (scheduled/dispatch job). Not built.

Reading this report as "the chat feature is injection-proof" would be wrong.
Read it as: **a stored injection payload cannot make the tool layer cross a
household boundary or leak PII**, and the model-behavior layers are named,
waived, and tracked — not silently skipped.
