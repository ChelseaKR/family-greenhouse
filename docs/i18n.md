# Internationalization

How UI strings work in this repo, and the gates that keep them honest. The
format decision (i18next-native JSON catalogs, no ICU MF1) is recorded in
[ADR 0007](adr/0007-i18n-json-catalogs-native-format.md); the portfolio-wide
requirements live in `STANDARDS/INTERNATIONALIZATION-STANDARD.md`.

## Where strings live

```
frontend/src/i18n/
├── index.ts                     # i18next bootstrap, language detection, shipping gate
├── format.ts                    # Intl.* date/number/currency helpers
├── coverage.ts                  # translated-share measurement (unit-test gate)
└── locales/
    ├── en/translation.json      # source locale — canonical key set
    └── es/
        ├── translation.json     # Spanish catalog (same keys, gate-enforced)
        └── translation.todo.json# untranslated/intentionally-equal markers
```

- Catalogs are i18next-standard per-locale namespace files
  (`<lng>/<namespace>.json`, single `translation` namespace today) and are the
  **single source of truth** for user-facing copy. Components render them via
  `useTranslation()` / `t('feature.key')`.
- Interpolation is i18next `{{var}}`; plurals are JSON-v4 CLDR suffixes.
  Always write explicit `_one` / `_other` (and whatever else the locale's
  CLDR rules require — Spanish also needs `_many`); never mix a bare key with
  suffixed forms of the same key.
- Dates, numbers, and currency never go through string concatenation — use
  `src/i18n/format.ts` (`Intl.*`, locale-aware).

## Adding or changing strings

1. Add the key to `en/translation.json` (English is the canonical key set).
2. Add the same key to **every** other locale. If you can't supply a real
   translation, copy the English value **and list the key in that locale's
   `translation.todo.json` under `todo`** — that is the untranslated-marker
   convention. Never machine-translate, and never leave an English value
   undeclared (the gate fails it as a silent fallback).
3. If a translation is genuinely identical to English (e.g. es `"No"`), list
   it under `intentionallyEqual` instead.
4. When a real translation lands, remove the key from `todo` — stale markers
   fail the gate too.

## Gates (merge-blocking; CI job `i18n Gates`, also in `npm run verify`)

Run locally with `npm run i18n:check` (repo root or `frontend/`).

| Check                                                                                                | Script                                         | Standard          |
| ---------------------------------------------------------------------------------------------------- | ---------------------------------------------- | ----------------- |
| Key parity — every logical key in every locale                                                       | `frontend/scripts/check-i18n-catalogs.mjs`     | G6                |
| Placeholder parity — identical `{{var}}` sets per key across locales                                 | same                                           | G5                |
| Plural categories — exactly the CLDR categories each locale requires                                 | same                                           | G5                |
| TODO markers — English-valued entries declared in `translation.todo.json`; no stale markers          | same                                           | (repo convention) |
| BCP-47 — locale dirs + `index.html` root `lang` well-formed/canonical                                | same                                           | G3                |
| UTF-8 — every git-tracked text file strict-decodes                                                   | same                                           | G1                |
| No new hardcoded JSX text — per-file ratchet against `frontend/scripts/i18n-hardcoded-baseline.json` | `frontend/scripts/check-hardcoded-strings.mjs` | G2                |

### The hardcoded-string ratchet

The scanner walks every JSX text node in `frontend/src` (curated long-form
content — blog, help, legal, care guides, changelog — is excluded as a
separate translation workstream). Existing debt (418 strings at baseline
time) is pinned per file; a file may never exceed its baseline, and when you
migrate strings to the catalog you must lower the baseline in the same PR
(`node scripts/check-hardcoded-strings.mjs --update-baseline` after reducing
— the diff shows counts only going down). `eslint-plugin-i18next`'s
`no-literal-string` additionally hard-errors for files enrolled in
`eslint.config.mjs` (attribute coverage: aria/alt/title), and files should be
enrolled there as they are fully migrated.

## Locales and shipping

`en` and `es`. Spanish is complete for real use (225/231 keys human-translated;
the 6 `todo`-marked entries are `_many` plural forms that only render for
counts ≥ 1,000,000) but stays behind the `VITE_ENABLE_NON_ENGLISH_LOCALES`
gate in `src/i18n/index.ts` until native-speaker review signs off — see the
coverage bar in `tests/unit/i18n/localeCoverage.test.ts`.

**Adding a locale:** create `locales/<bcp47-tag>/translation.json` with every
key (English values + full `todo` sidecar listing is acceptable as a scaffold),
add the tag to `ALL_LANGS` in `src/i18n/index.ts`, and — if it's RTL — to
`RTL_LANGS`. The gates enforce the rest from the first commit.
